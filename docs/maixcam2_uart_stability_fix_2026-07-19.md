# MaixCAM2 串口卡死问题：定位与修复总结（2026-07-19）

## 1. 结论

本轮已经验证有效的措施是：将 **STM32 → MaixCAM2** 的 `0x81` 控制帧周期由 **10 ms（100 Hz）** 降为 **100 ms（10 Hz）**。在当前接线和程序下，MaixCAM2 应用已连续稳定运行。

这是一项经过上板验证的**稳定性修复/缓解措施**，但不能据此断言根因已经百分之百确定为“100 Hz 的链路带宽不足”。更准确的结论是：问题只在 STM32 反向发送链路工作时出现，100 Hz 反向控制帧会触发 MaixCAM2 端的异常；降至 10 Hz 后异常消失。

当前建议将 10 Hz 作为调参阶段的默认值。该控制帧只同步工作模式、激光授权和圆周进度，不参与 IMU 速度内环或电机速度控制，因此 10 Hz 不影响云台的 200 Hz IMU 数据与视觉反馈闭环。

## 2. 现象与影响范围

### 2.1 表现

- MaixVision 图像停在某一帧，不再刷新；
- 随后 USB/MaixVision 与 MaixCAM2 的连接断开，表现为应用退出或设备重启；
- STM32 未上电时，相机程序稳定；
- 仅关闭 MaixCAM2 → STM32 的发送不能消除异常；
- 禁止 STM32 PB10 → MaixCAM2 A22 的反向链路后，相机稳定。

### 2.2 本轮未改动的部分

- MaixCAM2 → STM32 的视觉 `0x01` 与 IMU `0x02` 帧格式；
- IMU 读取频率 `200 Hz`；
- 视觉发送上限 `100 Hz`；
- UART4 / USART3 的 `115200 / 8N1` 配置；
- STM32 云台 PID、QD4310 速度模式和激光默认关闭策略。

## 3. 定位和修改顺序

| 顺序 | 操作与观察 | 得出的结论 |
| --- | --- | --- |
| 1 | 发现 MaixCAM2 在 STM32 上电、双向 UART 接通后会冻结并断开。 | 异常与联合运行有关，不能先归因于 YOLO 推理。 |
| 2 | 拔掉曾接在相机 UART2 上的 JY901S 后，曾观察到一次恢复。 | JY901S 的供电/串口连线可能造成过独立干扰，但不足以解释后续可重复的问题。 |
| 3 | 禁用相机端实际 `uart_dev.write()` 的发送路径后，问题仍会出现。 | 不能把根因简单归为 MaixCAM2 向 STM32 高频发送。 |
| 4 | STM32 改为 USART3 仅接收，PB10 配为高阻输入，并禁止反向 `0x81` 控制帧。 | 相机稳定，问题范围收敛到 STM32 → MaixCAM2 反向链路及其触发的相机端接收路径。该安全版本已提交为 `4c4c8cb`。 |
| 5 | 恢复 PB10 TX、USART3 双向模式和 100 Hz `0x81` 控制帧，准备验证双向链路。 | 保持协议不变，确认问题与反向帧频率相关。 |
| 6 | 将控制帧周期由 `10 ms` 调整为 `100 ms`，即 100 Hz 降为 10 Hz。 | 当前稳定运行，作为本轮采用的修复配置。 |

## 4. 前后方案 Diff

### 4.1 不稳定的双向配置

```text
STM32 USART3 TX (PB10) ──100 Hz，0x81 控制帧──> MaixCAM2 UART4 RX (A22)
MaixCAM2 UART4 TX (A21) ──视觉/IMU 观测帧──> STM32 USART3 RX (PB11)
```

MaixCAM2 当前程序中的 UART 使用方式：

- IMU 子线程每 `5 ms` 读取 IMU 并调用 `send_imu_frame()`；
- 视觉主循环调用 `poll_control_uart()` 读取/解析反向控制帧；
- 视觉主循环最多每 `10 ms` 调用 `send_vision_frame()`；
- 两个发送方通过 `uart_tx_lock` 防止两个写帧字节交错；
- 但 UART 的 `read()` 与另一线程的 `write()` 仍可能同时访问同一个 `uart_dev` 对象。

因此，100 Hz 的反向帧并不只是增加了字节数量；它会显著增加主视觉循环进入 `uart_dev.read()` 和协议解析的机会。在 MaixPy UART 驱动对跨线程读写的行为尚未完成独立验证前，应将这一并发访问视为高优先级风险。

### 4.2 当前稳定配置

```text
STM32 USART3 TX (PB10) ──10 Hz，0x81 控制帧───> MaixCAM2 UART4 RX (A22)
MaixCAM2 UART4 TX (A21) ──视觉/IMU 观测帧─────> STM32 USART3 RX (PB11)
```

STM32 端当前变更：

| 项目 | 修复前（本轮恢复测试） | 修复后（当前） |
| --- | ---: | ---: |
| USART3 工作模式 | `UART_MODE_TX_RX` | `UART_MODE_TX_RX` |
| PB10 | USART3_TX，推挽输出 | USART3_TX，推挽输出 |
| 反向控制帧开关 | `MAIXCAM_CONTROL_TX_ENABLE = 1U` | `MAIXCAM_CONTROL_TX_ENABLE = 1U` |
| 控制帧周期 | `10 ms` | `100 ms` |
| 控制帧频率 | `100 Hz` | `10 Hz` |
| 帧内容 | `Idle / 激光关闭 / 进度 0` | 不变 |

实际代码位置：

- `stm32/25Qdrive无刷电机/21F 5.22 （巡线单灰度版本） - 副本/21F/21F/Core/Src/usart.c`
- `stm32/25Qdrive无刷电机/21F 5.22 （巡线单灰度版本） - 副本/21F/21F/Hardware/usart3(maixcam)/maixcam.c`

## 5. 为什么不能简单归因于 115200 带宽

UART 为全双工；两个方向的带宽应分别估算。

| 方向 | 帧长 | 频率 | 线速占用（8N1） |
| --- | ---: | ---: | ---: |
| MaixCAM2 → STM32，IMU `0x02` | 21 B | 200 Hz | 约 42,000 bit/s，约 36% |
| MaixCAM2 → STM32，视觉 `0x01` | 18 B | 最多 100 Hz | 最多约 18,000 bit/s，约 16% |
| STM32 → MaixCAM2，控制 `0x81`（原） | 12 B | 100 Hz | 约 12,000 bit/s，约 10% |
| STM32 → MaixCAM2，控制 `0x81`（当前） | 12 B | 10 Hz | 约 1,200 bit/s，约 1% |

控制帧原本仅占其**反向方向**约 10% 的 115200 bit/s 线速，不足以单独证明“物理串口过载”。更合理的候选原因包括：

1. MaixCAM2 程序中同一 `uart_dev` 被 IMU 线程写入、视觉主循环读取/写入；写入之间有锁，但读写没有统一由一个线程串行化。
2. MaixPy UART 驱动在高频反向接收、视觉推理和 IMU 线程并存时出现阻塞、资源竞争或异常。
3. PB10→A22 的电平、供电、接地或布线抗干扰裕量不足。虽然已确认使用 3.3 V 逻辑和共地，仍应在需要恢复 100 Hz 前用示波器/逻辑分析仪复核波形。

## 6. 当前协议保持不变

统一帧仍为：

```text
55 AA | VER | TYPE | FLAGS | SEQ | LEN | PAYLOAD | CRC16_H CRC16_L
```

- `VER = 0x01`；
- CRC 为 CRC-16/CCITT-FALSE，覆盖 `VER` 到 `PAYLOAD`；
- 多字节整数与 IEEE-754 `float32` 均为大端序；
- `0x01` 视觉帧：两路 `float32` 像素误差与已应用控制序号；
- `0x02` IMU 帧：三路 `float32` 角速度（dps）；
- `0x81` 控制帧：`mode:uint8`、`circle_progress:uint16`；当前仅发送 Idle、激光关闭、进度 0。

本次仅修改 `0x81` 的发送周期，未修改字段、长度、校验或单位，因此 STM32 和 MaixCAM2 之间的协议兼容性保持不变。

## 7. 后续建议

1. 当前调参阶段保持 10 Hz 控制帧，并连续运行至少 30 分钟，记录 MaixVision 是否冻结、`CTRL` 是否持续绿色和 STM32 CRC/长度错误计数。
2. 若后续确实需要更快的 STM32→相机状态同步，优先实施并验证“唯一 UART 通信线程”：该线程独占 `uart_dev` 的读、解析、IMU 发送和视觉发送；视觉主循环只发布最新观测，不直接读写 UART。
3. 单线程架构稳定后，再按 10 Hz → 20 Hz → 50 Hz → 100 Hz 逐级恢复反向控制帧。每一级至少运行 10 分钟再升级。
4. 若单线程架构下仍复现异常，应停止继续提升频率，转而检查 PB10/A22 波形、共地、电源纹波、串口线长度及电机驱动产生的共模干扰。
