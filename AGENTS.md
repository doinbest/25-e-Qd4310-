# AGENTS.md

## 1. 项目定位

本仓库用于电子设计竞赛控制方向的嵌入式项目。

系统通常由以下部分组成：

```text
摄像头、陀螺仪、编码器、测距模块等传感器
                        ↓
                 单片机主控制器
                        ↓
          控制算法、模式管理和安全判断
                        ↓
      直流电机、无刷电机、步进电机、舵机
```

基本原则：

* 单片机是系统主控制器。
* 摄像头属于传感器或感知节点。
* 摄像头负责提取目标信息并反馈给单片机。
* 单片机负责模式管理、控制决策、安全判断和执行器控制。
* 电机或舵机是系统最终执行机构。
* 除非任务明确要求，否则摄像头不直接决定电机输出。
* 系统功能应优先保证正确、稳定、可读和便于现场调试。

本项目优先级为：

```text
正确性
> 安全性
> 可读性
> 可调试性
> 实时性
> 通用性
```

代码应优先采用：

> 简单、直接、稳定、容易理解、容易上板调试的实现。

不要为了未来可能出现的需求增加当前项目不需要的复杂度。

## 3. 系统角色划分

### 3.1 单片机职责

单片机通常负责：

* 读取传感器数据
* 接收视觉识别结果
* 管理比赛工作模式
* 判断数据是否有效
* 判断通信是否超时
* 执行 PID 或其他控制算法
* 控制电机、舵机和其他执行器
* 处理异常和安全停止
* 管理整个系统运行状态

模式状态、控制状态和安全状态原则上由单片机维护。

### 3.2 摄像头职责

摄像头通常负责：

* 图像采集
* 目标检测
* 目标筛选
* 坐标计算
* 必要的视觉滤波
* 数据有效性判断
* 向单片机发送识别结果

摄像头输出的数据可以包括：

* 是否识别到目标
* 目标中心坐标
* 目标与参考点的误差
* 目标大小
* 目标角度
* 目标类别
* 识别置信度
* 帧序号或时间信息

具体字段根据当前项目和用户任务确定。

### 3.3 职责边界

除非任务明确要求，摄像头端不要：

* 直接输出电机电流
* 直接输出电机 PWM
* 直接维护整车或云台的完整工作模式
* 代替单片机完成最终安全决策
* 代替单片机执行完整电机控制闭环
* 在通信中隐藏坐标单位和方向定义

推荐发送观测结果：

```text
valid = 1
error_x = -12
error_y = 8
```

不推荐直接发送最终执行量：

```text
left_motor_pwm = 638
right_motor_pwm = 412
```

## 4. 项目信息获取原则

本文件不固定具体硬件型号、引脚和通信协议。

处理任务时，应根据当前项目获取实际信息。

信息来源优先级：

1. 用户当前任务描述
2. 当前仓库代码
3. 芯片配置文件
4. STM32 `.ioc` 文件
5. MaixVision 工程代码
6. 项目内设备手册
7. 项目内协议文档
8. 接线说明
9. 调试日志

不得仅凭经验猜测：

* MCU 完整型号
* GPIO 引脚
* 外设实例
* 定时器通道
* 串口波特率
* CAN 波特率
* 电机 ID
* 设备地址
* 通信帧格式
* 数据大小端
* 坐标方向
* 控制方向
* 数据单位
* 数值缩放比例

如果无法从代码和文档中确认，应明确标记为：

```text
尚未确认
```

不得将推测描述为已经确认的事实。

## 5. 用户指令优先级

用户当前任务中的明确要求高于默认工作流程。

例如：

* 用户要求“只分析”时，不修改文件。
* 用户要求“先给方案”时，先输出实施方案。
* 用户要求“直接修改”时，完成必要理解后直接实施。
* 用户要求“给完整代码”时，提供完整代码。
* 用户要求“只改一个模块”时，不扩展到其他模块。
* 用户要求保留某项功能时，不得为了优化而删除。
* 用户提供新的协议、参数或方向定义时，以最新要求为准。
* 用户明确禁止修改某个文件时，不得修改该文件。

本文件用于提高项目理解和代码质量，不应妨碍完成用户明确提出的任务。

## 6. 首次读取项目规则

当本次会话首次读取当前项目，或者尚不了解项目结构时，应先建立项目概览。

需要确认：

1. 当前项目属于哪种平台：

   * STM32
   * MSPM0G3507
   * MaixCAM2
   * 多端联合项目

2. 程序入口文件在哪里。

3. 主要文件和模块分别负责什么。

4. 当前系统完整数据流是什么，例如：

```text
摄像头、陀螺仪或其他传感器
→ 数据采集与处理
→ 通信
→ 单片机主控制器
→ 控制算法和模式判断
→ 电机或舵机执行
```

5. 当前使用哪些通信方式。

6. 通信数据在哪里生成、发送、接收和解析。

7. 哪些参数需要现场调节。

8. 哪些文件属于：

   * 自动生成代码
   * 芯片厂商库
   * 用户业务代码
   * 配置文件
   * 通信协议文件

9. 当前代码是否存在明显问题，例如：

   * 过长函数名
   * 过长宏名称
   * 不必要的指针参数
   * 不必要的抽象层
   * 含义不清楚的变量
   * 隐藏主流程的多层封装
   * 分散且没有说明的关键参数

首次项目概览的目的是建立正确上下文，不代表必须停止当前任务。

如果用户已经明确要求修改代码，则完成必要理解后继续实施。

如果用户明确要求“只分析”“先不要修改”或“先给方案”，则只输出分析结果，不修改文件。

## 7. 修改任务开始前检查

每次准备修改代码前，应先确认以下内容。

### 7.1 当前任务

* 用户真正要求解决的问题是什么。
* 当前现象是什么。
* 期望结果是什么。
* 如何判断修改成功。
* 哪些功能必须保留。
* 哪些内容禁止修改。

### 7.2 当前平台

* 当前代码是 C 还是 Python。
* 当前平台是 STM32、MSPM0G3507 还是 MaixCAM2。
* 当前使用的库和 API 是什么。
* 当前构建或运行方式是什么。

### 7.3 数据流

* 输入来自哪里。
* 数据在哪里采集。
* 数据经过哪些处理。
* 数据在哪里发送。
* 单片机在哪里接收。
* 控制算法在哪里执行。
* 电机命令在哪里产生。
* 异常情况下如何停止。

### 7.4 协议与单位

* 通信协议是否已经确认。
* 字段顺序是否明确。
* 数据类型是否明确。
* 有符号数和无符号数是否明确。
* 字节顺序是否明确。
* 坐标方向是否明确。
* 控制方向是否明确。
* 单位和缩放比例是否明确。
* 控制周期是否明确。

不确定的信息不得猜测。

如果不确定的信息会直接影响代码正确性，应：

1. 查找当前项目中的代码、配置和文档。
2. 根据已确认信息完成安全的部分。
3. 对无法确认的部分添加明确说明。
4. 不伪造设备协议和硬件参数。

## 9. 基本开发原则

处理代码任务时必须遵守：

1. 先阅读现有代码，再进行修改。
2. 优先进行最小范围修改。
3. 不修改与当前任务无关的功能。
4. 不重写已经正常工作的模块。
5. 不进行没有明确收益的大规模重构。
6. 不主动引入 RTOS。
7. 不主动引入队列、互斥量和复杂事件系统。
8. 不主动使用动态内存。
9. 不主动引入新的第三方库。
10. 不随意改变通信协议。
11. 不随意改变坐标和控制方向。
12. 不同时调整多个无关参数。
13. 优先沿用项目已有且合理的代码风格。
14. 修改后检查是否影响已有功能。
15. 不为了代码显得高级而增加复杂度。

不要顺便进行：

* 无关格式化
* 无关重命名
* 无关目录调整
* 无关参数优化
* 无关文件迁移
* 无关架构重构

代码需要让参赛队员快速理解和修改。

一名熟悉单片机和基础 Python 的队员，应能够较快找到：

1. 图像或传感器数据在哪里采集。
2. 模型或识别算法在哪里运行。
3. 识别结果在哪里筛选。
4. 坐标和误差在哪里计算。
5. 数据在哪里发送。
6. 单片机在哪里接收。
7. 控制算法在哪里执行。
8. 电机命令在哪里产生。
9. 参数在哪里调整。
10. 异常时如何停止。

主要流程不应隐藏在：

* 多层回调
* 多级函数指针
* 通用上下文对象
* 复杂宏
* 装饰器链
* 大量包装函数
* 不必要的类层级
* 动态注册系统

## 11. 通用命名规范

### 11.1 基本形式

函数和变量使用小写下划线：

```text
target_x
error_y
motor_speed
uart_parse_frame
control_update
```

常量使用大写下划线：

```text
CONTROL_PERIOD_MS
UART_RX_SIZE
TARGET_TIMEOUT_MS
```

类型名称可以使用大驼峰：

```text
VisionTarget
ControlMode
MotorState
```

### 11.2 函数名称

函数名建议使用 2～4 个单词。

推荐：

```c
motor_set_speed();
motor_stop();
uart_parse_frame();
target_calc_error();
control_update();
```

推荐：

```python
detect_target()
calc_error()
send_result()
draw_debug()
```

不推荐：

```text
gimbal_motor_closed_loop_velocity_controller_update_handler
vision_target_coordinate_error_calculation_and_filter_process
```

应简化为：

```text
gimbal_update
target_calc_error
```

禁止使用含义模糊的名称：

```text
do_work
handle_data
process_all
run_task
func1
```

### 11.3 变量名称

推荐：

```text
target_x
target_y
error_x
error_y
motor_speed
frame_length
timeout_ms
```

不推荐：

```text
a
b
tmp1
data2
current_received_target_horizontal_coordinate_value
```

循环变量可以使用：

```text
i
j
index
count
```

同一概念在整个项目中应使用相同名称。

不要混用：

```text
err_x
x_error
horizontal_error
target_offset_x
```

应统一为一种名称，例如：

```text
error_x
```

### 11.4 单位名称

涉及物理量时，在变量名中体现单位：

```text
timeout_ms
period_us
speed_rpm
angle_deg
current_ma
voltage_mv
distance_mm
radius_px
```

协议原始值使用 `_raw`：

```text
speed_raw
angle_raw
current_raw
```

不要让同一个变量有时表示原始值，有时表示实际物理量。

### 11.5 缩写

允许使用常见嵌入式缩写：

* UART
* CAN
* SPI
* I2C
* ADC
* PWM
* DMA
* PID
* RPM
* GPIO
* FPS
* ROI
* NMS

不要自行创造难以理解的缩写。

## 12. C 代码规范

### 12.1 语言标准

单片机代码默认使用 C11。

优先使用明确宽度整数类型：

```c
uint8_t
int16_t
uint16_t
int32_t
uint32_t
```

开关状态优先使用 `bool` 或明确枚举。

避免无理由混用：

* `int`
* `long`
* 有符号类型
* 无符号类型

类型转换必须有明确原因，并确认数值范围安全。

### 12.2 指针

只有确实必要时使用指针。

合理场景：

* HAL 或芯片 SDK 接口要求
* 数组和缓冲区
* 修改结构体状态
* 传递较大的结构体
* 返回多个结果
* 中断回调
* 底层寄存器访问

简单标量参数优先按值传递。

推荐：

```c
void motor_set_speed(int16_t speed_rpm);
float pid_update(float error);
void laser_set_state(bool enabled);
```

不推荐：

```c
void motor_set_speed(const int16_t *speed_rpm);
void pid_update(float *error, float *output);
void laser_set_state(bool *enabled);
```

只读指针使用 `const`：

```c
bool uart_parse_frame(const uint8_t *buffer, uint16_t length);
```

需要修改对象时使用非 `const` 指针：

```c
void pid_reset(PidController *pid);
```

除非底层接口要求，否则避免：

* `void *`
* 二级指针
* 函数指针表
* 动态分配指针
* 通用上下文指针
* 用指针隐藏全局状态

不要为了接口形式统一而将所有参数改成指针。

### 12.3 宏

宏主要用于：

* 编译开关
* 缓冲区大小
* 控制周期
* 通信常量
* 控制限幅
* 现场调试参数

推荐：

```c
#define CONTROL_PERIOD_MS       10U
#define UART_RX_SIZE            64U
#define MOTOR_MAX_SPEED_RPM     800
#define TARGET_TIMEOUT_MS       200U
```

宏名建议使用 2～5 个单词。

不推荐：

```c
#define GIMBAL_HORIZONTAL_AXIS_MOTOR_MAXIMUM_ALLOWED_ROTATIONAL_SPEED_VALUE 800
```

推荐：

```c
#define YAW_MAX_SPEED_RPM 800
```

禁止使用宏隐藏：

* 多条语句
* 复杂计算
* 循环
* 状态修改
* 指针操作
* 多次硬件调用

复杂逻辑使用普通函数或简单的 `static inline` 函数。

### 12.4 函数

一个函数完成一个清晰任务。

推荐：

```c
uart_receive_data();
uart_parse_frame();
control_update();
motor_send_command();
```

不推荐：

```c
receive_parse_filter_pid_control_send_and_debug();
```

普通函数建议控制在约 20～60 行。

函数明显超过约 80 行时，应检查是否承担了多个独立职责。

但不要机械拆分成大量：

* 只有两三行
* 只调用一次
* 不能降低复杂度
* 只是转调另一个函数

的辅助函数。

函数参数通常不超过 4 个。

不要仅为了减少参数数量而创建复杂的一次性结构体。

### 12.5 返回值

只有一个结果时，优先使用返回值：

```c
float pid_update(float error);
bool target_is_valid(void);
```

需要返回多个结果时，可以使用输出指针：

```c
bool target_get_error(int16_t *error_x, int16_t *error_y);
```

## 13. Python 代码规范

### 13.1 基本风格

Python 使用清晰、直接的过程式实现。

优先使用：

* 普通函数
* 简单列表和元组
* 简单字典
* 少量清晰状态
* 明确主循环

不要为了形式上的高级感主动使用：

* 元类
* 多重继承
* 复杂装饰器
* 生成器链
* 过度函数式写法
* 大量 lambda
* 动态属性注入
* 插件注册系统
* 复杂事件框架

### 13.2 类的使用

只有多个状态和行为确实属于同一对象时才使用类。

适合使用类的情况：

* PID 控制器
* 串口协议解析器
* 具有持续状态的目标跟踪器
* 设备驱动本身以类形式提供

不要为以下简单功能单独创建类：

* 单次坐标计算
* 简单限幅
* 单次数据发送
* 单个状态变量
* 只有一个函数的工具类

### 13.3 函数设计

推荐：

```python
def detect_target(img):
    ...

def calc_error(target_x, target_y):
    ...

def send_result(valid, error_x, error_y):
    ...
```

不推荐：

```python
def process_capture_detection_filter_coordinate_transform_result_pack_send_draw():
    ...
```

函数参数尽量不超过 4 个。

复杂数据确实属于同一结果时，可以使用简单字典、元组或对象，但不要层层嵌套。

### 13.4 避免晦涩写法

不推荐：

```python
result = max(min(kp * err + ki * total + kd * (err - last) / dt, limit), -limit)
```

推荐：

```python
p_output = kp * error
i_output = ki * integral
d_output = kd * derivative

output = p_output + i_output + d_output
output = limit_value(output, -output_limit, output_limit)
```

不要为了减少代码行数牺牲可读性。

## 14. 控制流程规范

嵌套层级尽量不超过 3 层。

不推荐：

```c
if (enabled)
{
    if (target_valid)
    {
        if (!timeout)
        {
            if (motor_ready)
            {
                motor_set_speed(speed);
            }
        }
    }
}
```

推荐：

```c
if (!enabled)
{
    motor_stop();
    return;
}

if (!target_valid || timeout || !motor_ready)
{
    motor_stop();
    return;
}

motor_set_speed(speed);
```

复杂表达式应拆成有意义的中间变量。

但不要滥用提前返回，使简单流程变得零散。

---

## 15. 结构体、枚举与数据对象

只有一组数据确实属于同一对象时，才使用结构体。

推荐：

```c
typedef struct
{
    int16_t error_x;
    int16_t error_y;
    bool valid;
    uint32_t update_time_ms;
} VisionTarget;
```

不要创建只有一个简单字段的结构体：

```c
typedef struct
{
    int16_t speed;
} MotorSpeedValue;
```

模式和状态优先使用枚举：

```c
typedef enum
{
    MODE_IDLE = 0,
    MODE_TRACK,
    MODE_LOCK
} ControlMode;
```

不要在业务代码中直接使用无含义数字：

```c
if (mode == 3)
{
}
```

应使用：

```c
if (mode == MODE_LOCK)
{
}
```

结构体和 Python 数据结构不要嵌套过深。

## 16. 文件与模块规范

小型电赛项目优先保持少量清晰模块。

单片机端可以采用：

```text

control.c
control.h
motor.c
motor.h
uart_protocol.c
uart_protocol.h
pid.c
pid.h
config.h
```

视觉端可以采用：

```text
main.py
vision.py
uart_protocol.py
config.py
```

模块职责示例：

* `main.c`：初始化和主循环调度
* `control.c`：主要控制流程和模式管理
* `motor.c`：电机命令和反馈
* `uart_protocol.c`：串口协议接收和解析
* `pid.c`：PID 基础计算
* `main.py`：视觉程序入口和主循环
* `vision.py`：识别、筛选和坐标计算

```text

services/
providers/
contexts/
factories/
managers/
```

除非当前项目已经使用这些结构，并且继续沿用能够明显提升清晰度。

每个模块应能用一句话解释其职责。

## 17. MaixCAM2 视觉代码规范

### 17.1 初始化与主循环分离

模型、摄像头、显示器和通信对象只初始化一次。

不要在每帧循环中重复：

* 加载模型
* 创建摄像头对象
* 打开串口
* 创建显示对象
* 读取配置文件
* 分配大型固定缓冲区

推荐结构：

```python
def init_camera():
    ...

def init_model():
    ...

def detect_target(img):
    ...

def send_result(valid, error_x, error_y):
    ...

def main():
    camera = init_camera()
    model = init_model()

    while True:
        img = camera.read()
        target = detect_target(img)
        ...
```

### 17.2 视觉处理流程

视觉流程应清晰：

```text
采集图像
→ 模型推理或传统识别
→ 筛选目标
→ 计算目标坐标
→ 计算反馈数据
→ 发送给单片机
→ 必要的屏幕显示
```

不要把推理、筛选、通信和绘图全部混在一个巨大循环中。

但也不要拆成大量没有实际意义的小函数。

### 17.3 YOLO 识别

修改 YOLO 相关代码前，应确认：

* 模型路径
* 输入尺寸
* 标签列表
* 置信度阈值
* NMS 阈值
* 图像格式
* 坐标缩放方式
* 目标筛选规则

不要未经确认修改：

* 模型输入尺寸
* 类别索引
* 标签顺序
* 坐标映射
* 图像裁剪区域
* 推理频率

### 17.4 目标筛选

存在多个目标时，应明确筛选规则，例如：

* 置信度最高
* 距离画面中心最近
* 面积最大
* 与上一帧位置最近
* 指定类别

不要默认检测结果中的第一个目标一定是正确目标，除非 API 明确保证排序。

### 17.5 性能要求

每帧循环中避免：

* 不必要的整图格式转换
* 不必要的图像复制
* 重复创建大型数组
* 重复计算固定参数
* 大量屏幕文字绘制
* 每帧持续打印大量信息
* 阻塞式长时间等待
* 不必要的高分辨率处理

只需要 ROI 时，优先处理 ROI。

固定矩阵、阈值和映射参数应尽量在循环外准备。

性能优化不得删除用户明确要求保留的识别功能。

### 17.6 帧率定义

显示帧率时，应明确是哪一种：

* 摄像头采集帧率
* 模型推理帧率
* 视觉处理帧率
* 完整主循环帧率
* 通信发送频率

默认显示完整主循环帧率，除非用户明确要求显示其他帧率。

不要用模型单次推理帧率冒充整体运行帧率。

## 18. 裸机主循环规范

单片机裸机主循环应保持清晰。

推荐：

```c
while (1)
{
    communication_update();
    sensor_update();
    control_update();
    motor_update();
    debug_update();
}
```

需要固定周期时，使用时间戳调度：

```c
if (now_ms - last_control_ms >= CONTROL_PERIOD_MS)
{
    last_control_ms = now_ms;
    control_update();
}
```

避免在主循环中使用长时间阻塞延时。

需要等待时，优先使用：

* 时间戳
* 简单状态变量
* 非阻塞状态机
* 周期调度

状态机应保持直观，不要引入复杂状态框架。

---

## 19. 中断规范

中断中只执行必要操作：

* 保存接收数据
* 更新接收索引
* 设置标志位
* 更新时间戳
* 调用轻量级底层接口

中断中禁止：

* 阻塞延时
* 长循环
* 大量打印
* 完整 PID 计算
* 复杂浮点计算
* 等待通信发送完成
* 动态内存分配
* 执行完整业务流程

复杂解析和控制计算放在主循环或周期函数中。

中断和主循环共享变量时，应考虑：

* `volatile`
* 数据一致性
* 更新顺序
* 原子访问

不要无理由将所有全局变量声明为 `volatile`。

---

## 20. 视觉与单片机通信规范

视觉端与单片机之间的通信应简单、稳定、容易调试。

修改协议前必须确认：

* 帧头和帧尾
* 数据字段
* 数据长度
* 数据类型
* 有符号数
* 字节顺序
* 单位
* 缩放比例
* 校验方式
* 发送频率
* 超时条件

视觉反馈字段应使用明确名称：

```text
valid
target_x
target_y
error_x
error_y
angle_deg
confidence
```

不要使用：

```text
data1
data2
value
temp
flag2
```

单片机必须能够判断视觉数据是否过期。

视觉端停止发送、识别失败或通信异常时，单片机不能长期使用旧数据。

通信代码应考虑：

* 半包
* 粘包
* 丢字节
* 非法数据
* 缓冲区越界
* 接收超时
* 重复帧
* 数据更新频率

通信协议修改时，必须同时检查视觉端和单片机端，不能只修改其中一端。

---

## 21. 控制算法规范

单片机负责最终控制决策。

涉及 PID、滤波、误差计算和电机控制时，应明确：

* 输入单位
* 输出单位
* 控制周期
* 坐标方向
* 电机正方向
* 输出限幅
* 积分限幅
* 死区
* 通信超时
* 目标丢失处理

不要在同一次修改中同时改变：

* 通信协议
* 坐标定义
* PID 参数
* 滤波方法
* 控制周期
* 电机输出范围

除非任务明确要求同时修改。

视觉端可以进行必要的抖动抑制，但不要加入明显增加控制延迟的复杂预测或滤波，除非用户明确要求。

## 22. 执行机构规范

最终执行机构通常为：

* 直流减速编码器电机
* 无刷电机
* 步进电机
* 舵机

不同电机不应强行共用复杂的通用驱动框架。

根据当前实际设备使用清晰接口。

直流电机示例：

```c
motor_set_pwm(MOTOR_LEFT, left_pwm);
motor_set_pwm(MOTOR_RIGHT, right_pwm);
```

无刷云台示例：

```c
gimbal_set_speed(yaw_speed, pitch_speed);
```

步进电机示例：

```c
stepper_set_target(target_steps);
```

舵机示例：

```c
servo_set_angle(angle_deg);
```

只有当前项目确实存在多个同类设备时，才设计共用接口。

不要为少量固定设备设计动态注册系统。

## 23. STM32 自动生成代码规则

处理 STM32CubeMX 工程时：

* 不修改 STM32 HAL 官方库文件。
* 不修改 `Drivers` 中的官方代码。
* 不删除 CubeMX 注释标记。
* 用户代码优先放在 `USER CODE BEGIN/END` 区域。
* 新功能优先放入独立 `.c` 和 `.h` 文件。
* 不重新生成 `.ioc`，除非任务明确要求。
* 不随意更改已正常工作的时钟配置。
* 不随意更改已正常工作的引脚和外设配置。

例如：

```c
/* USER CODE BEGIN 2 */

/* USER CODE END 2 */
```

新建源文件后，应确认已经加入 EIDE 构建。

如果必须修改 CubeMX 生成区域，应说明该修改可能在重新生成工程时丢失。

## 24. MSPM0G3507 项目规则

处理 MSPM0G3507 项目时：

* 沿用当前 SDK 和驱动接口。
* 先阅读已有初始化代码。
* 不套用 STM32 HAL 函数。
* 不自行替换整套底层驱动。
* 不修改厂商库文件，除非任务明确要求。
* 不随意修改启动文件。
* 不随意修改链接脚本。
* 不随意修改时钟和外设配置。

平台相关代码可以与业务控制逻辑适度分离，但不要建立复杂跨平台抽象层。

## 25. 配置参数规范

需要现场调节的参数应集中在：

* `config.h`
* `config.py`
* 对应模块文件顶部
* 明确标记的参数区域

C 示例：

```c
#define CONTROL_PERIOD_MS       10U
#define TARGET_TIMEOUT_MS       200U
#define MOTOR_MAX_SPEED_RPM     800

#define YAW_KP                  1.2f
#define YAW_KI                  0.0f
#define YAW_KD                  0.08f
```

Python 示例：

```python
CONFIDENCE_THRESHOLD = 0.50
TARGET_TIMEOUT_MS = 200
UART_SEND_INTERVAL_MS = 20
DEBUG_DRAW_ENABLE = True
```

禁止将关键参数分散在大量函数内部。

避免没有名称和说明的魔法数字。

关键参数应说明调整影响：

```c
// 增大后响应更快，但过大可能导致振荡。
#define YAW_KP 1.2f
```

## 26. 注释规范

注释主要说明：

* 为什么这样实现
* 数据单位
* 坐标方向
* 电机正方向
* 硬件限制
* 协议依据
* 参数调整效果
* 特殊边界情况

推荐：

```c
// 图像坐标向右为正，但云台 yaw 正方向相反，因此取负号。
yaw_speed = -pid_update(&yaw_pid, error_x);
```

推荐：

```python
# 发送最新识别结果，避免单片机继续使用过期坐标。
send_result(valid, error_x, error_y)
```

不推荐：

```c
// 设置速度
motor_set_speed(speed);
```

不要为每行简单代码添加重复注释。

已有清晰中文注释应尽量保留。

新增注释优先使用中文，常见专业术语可以保留英文。

## 27. 调试输出规范

调试信息应可以统一开启或关闭。

单片机端：

```c
#define DEBUG_ENABLE 1
```

视觉端：

```python
DEBUG_PRINT_ENABLE = True
DEBUG_DRAW_ENABLE = True
```

调试信息使用明确前缀：

```text
[CAM]
[YOLO]
[UART]
[CTRL]
[MOTOR]
[ERROR]
```

不要：

* 在高速中断中持续打印。
* 每帧打印大量重复数据。
* 输出没有名称的数字。
* 让显示或打印明显降低控制性能。

调试显示和调试打印不得改变实际发送给单片机的数据。

## 28. 安全规则

以下情况应进入安全状态：

* 视觉目标丢失
* 视觉数据超时
* 串口数据非法
* 电机反馈超时
* 编码器数据异常
* 控制输出超过限制
* 系统刚上电
* 模式切换未完成

安全状态通常包括：

* 电机输出归零
* 电机缓慢停止
* 清除必要的 PID 积分
* 禁止继续使用过期数据
* 激光默认关闭
* 舵机或云台保持安全位置

不要删除已有的：

* 输出限幅
* 超时保护
* 目标丢失保护
* 电机停止逻辑
* 上电保护

未经明确要求，不要让执行器在上电后自动执行危险动作。

## 29. 禁止过度抽象

除非任务明确要求，否则不要主动使用：

### C 端

* 面向对象风格 C 封装
* 函数指针模拟虚函数
* 通用设备注册表
* 多层回调
* 事件总线
* 消息队列
* 工厂模式
* 依赖注入
* 通用生命周期管理
* 复杂跨平台接口层

### Python 端

* 多层类继承
* 元类
* 复杂装饰器
* 插件框架
* 动态注册系统
* 复杂异步框架
* 大量生成器和 lambda
* 为简单流程创建多个管理器类

如果一个函数只是调用另一个函数，并且没有增加：

* 安全检查
* 单位转换
* 参数限制
* 可读性

则通常不需要该包装函数。

不要创建只使用一次、且不能明显降低复杂度的抽象。

## 30. 可读性检查

生成或修改代码后，应检查：

* 是否存在过长函数名。
* 是否存在过长宏名称。
* 是否存在不必要的指针。
* 是否存在不必要的类。
* 是否存在不必要的结构体。
* 是否存在过多函数参数。
* 是否存在超过 3 层的嵌套。
* 是否存在复杂多行宏。
* 是否存在晦涩的一行表达式。
* 是否存在无意义包装函数。
* 是否存在只使用一次的抽象。
* 是否存在分散的关键参数。
* 是否存在不明确的单位。
* 是否存在不明确的坐标方向。
* 是否能直接看懂完整数据流。

如果某种设计只是更加通用，但明显降低可读性，应选择更简单的实现。

## 31. 编译与验证

### 31.1 单片机代码

优先使用当前 EIDE 工程已有的构建配置。

检查：

* 编译是否通过
* 是否产生新警告
* 新文件是否加入构建
* 是否存在未使用变量
* 是否存在隐式函数声明
* 是否存在类型不匹配
* 是否存在数组越界风险
* 是否存在有符号与无符号混用
* RAM 和 Flash 是否异常增加

如果无法实际编译，应明确说明：

```text
当前环境未配置对应工具链，因此只完成了静态代码检查，
未完成实际编译验证。
```

### 31.2 Python 代码

检查：

* Python 语法是否正确
* 模块导入是否正确
* API 名称是否与当前平台一致
* 模型路径是否正确
* 串口初始化是否正确
* 主循环是否存在明显阻塞
* 是否每帧重复初始化对象
* 异常时是否能够发送无效状态
* 屏幕绘制是否明显影响帧率

### 31.3 联合验证

视觉端与单片机联合修改时，检查：

* 双方协议是否一致
* 字段顺序是否一致
* 数据类型是否一致
* 有符号数是否一致
* 单位是否一致
* 坐标方向是否一致
* 发送频率是否合理
* 单片机超时是否生效
* 目标丢失时执行机构是否安全停止

如果无法连接硬件，应明确说明：

```text
当前未进行硬件测试，通信方向、执行器方向和实际控制效果
仍需上板验证。
```

不得伪造：

* 编译成功
* 烧录成功
* 串口通信成功
* 电机运行正常
* 识别帧率
* 上板测试结果
